YARM KERNEL CODE REVIEW
=======================
Project: YARM (Yet Another Rust Microkernel)
Language: Rust (no_std, bare-metal)
Targets: Embedded, aerospace, automotive (RISC-V, x86_64, AArch64, ...)
Architecture: Capability-based microkernel, userspace servers for all services

This document contains actionable fix instructions for each reviewed module.
Severity: CRITICAL > HIGH > MEDIUM > LOW
Format per issue: SEVERITY | FILE | ISSUE | FIX


================================================================================
MODULE: ipc.rs
================================================================================

CRITICAL | ipc.rs | send() returns Err(Message) instead of typed error
The Endpoint::send() method returns Result<(), Message> on queue-full instead
of Result<(), IpcError>. This is inconsistent with every other fallible function
and prevents use of the ? operator.
FIX: Add IpcError::EndpointFull variant. Change send() signature to
Result<(), IpcError>. If the caller needs the message back on failure, use
Result<(), (IpcError, Message)>.

CRITICAL | ipc.rs | MAX_ENDPOINT_DEPTH power-of-two invariant unguarded
The ring buffer uses bitmask indexing: (head + len) & (MAX_ENDPOINT_DEPTH - 1)
which is only correct when MAX_ENDPOINT_DEPTH is a power of two. There is no
compile-time check enforcing this.
FIX: Add:
  const _: () = assert!(MAX_ENDPOINT_DEPTH.is_power_of_two(),
    "MAX_ENDPOINT_DEPTH must be a power of two for bitmask indexing");

HIGH | ipc.rs | unpack_register_payload ignores len in output
The len parameter only gates the None return path. The output array is always
fully populated regardless of len. Callers passing len=4 still get all
IPC_REGISTER_BYTES populated.
FIX: Either return (array, len) tuple so the meaningful byte count is conveyed,
or document explicitly that len is a validity bound only, not a trim hint.

HIGH | ipc.rs | pack_register_payload silently truncates oversized payload
If payload.len() > IPC_REGISTER_BYTES the excess bytes are dropped silently.
FIX: Return Result<[usize; IPC_REGISTER_WORDS], IpcError> and return
Err(IpcError::PayloadTooLarge) when payload is too long.

HIGH | ipc.rs | Redundant compile-time assert catches nothing
  const _: [(); IPC_REGISTER_WORDS] = [(); syscall_abi::IPC_REGISTER_WORDS];
IPC_REGISTER_WORDS IS syscall_abi::IPC_REGISTER_WORDS so this always passes.
FIX: Remove or replace with a meaningful invariant check.

HIGH | ipc.rs | SharedMemoryRegion placement couples IPC to memory
SharedMemoryRegion encode/decode belongs in a memory or capability module, not
ipc.rs. IPC should pass opaque capability handles, not know about memory regions.
FIX: Move SharedMemoryRegion to vm.rs or cap.rs. IPC layer passes CapId.

MEDIUM | ipc.rs | while loops instead of for ranges (recurring pattern)
All loops using while i < N { ...; i += 1 } should use for i in 0..N.
FIX: Replace all while-counter loops with idiomatic for range loops throughout.

MEDIUM | ipc.rs | IpcError missing Display impl
FIX: Implement core::fmt::Display for IpcError with human-readable messages.

MEDIUM | ipc.rs | Message::MAX_PAYLOAD = 56 is undocumented magic number
FIX: Add a doc comment explaining where 56 comes from (register budget minus
header fields) or derive it from a constant expression.

LOW | ipc.rs | SharedMemoryRegion::decode could be const fn
encode is const fn but decode is not. The manual byte loop pattern from encode
can be applied to decode for symmetry.
FIX: Rewrite decode using manual byte-by-byte indexing to make it const fn.


================================================================================
MODULE: capabilities.rs
================================================================================

CRITICAL | capabilities.rs | revoke() silently drops children when stack fills
The iterative revoke uses a fixed-size stack array. When the stack fills during
child enumeration, remaining children are silently skipped and never revoked.
This leaves live capabilities with dangling parent references — a security hole.
FIX: Replace the stack-based traversal with a mark-and-sweep approach:
  1. Allocate a [bool; MAX_CAPABILITIES] marked array, set marked[id.index()] = true
  2. Loop: for each slot, if entry.parent is marked and slot not marked, mark it, set changed=true
  3. Repeat until !changed
  4. Sweep: for all marked slots, clear entry and increment generation

CRITICAL | capabilities.rs | mint() returns Err(Capability) instead of typed error
Same pattern as ipc.rs send(). Forces callers to pattern-match a raw Capability
to detect "space full". Makes ? unusable.
FIX: Add CapabilityDeriveError::SpaceFull (or rename to CapError). Change
mint() to return Result<CapId, CapabilityDeriveError>.

CRITICAL | capabilities.rs | name: &'static str in Capability struct
Every Capability carries 16 bytes (pointer + length) for a debug name. In a
microkernel, capabilities are opaque kernel objects. Names are a userspace
debugging concern. With 128 slots this wastes 2KB and couples the kernel to
string literals.
FIX: Remove the name field from Capability entirely. Remove it from Capability::new().
If debug names are needed, use a cfg(debug_assertions) side table keyed by CapId.

HIGH | capabilities.rs | NO_TRANSFER_CAP sentinel excludes valid cap ID u64::MAX
CapId(u64::MAX) is a theoretically valid capability ID but is reserved as sentinel.
FIX: Add a comment explicitly documenting the constraint. Consider a NonMaxU64
newtype or use a separate bool field to indicate "no transfer".

HIGH | capabilities.rs | DmaRegion uses usize for offset and len
usize is 32 bits on 32-bit targets. DMA offsets and lengths should be u64.
FIX: Change DmaRegion { id: u64, offset: u64, len: u64 }

HIGH | capabilities.rs | revoke() returns bool instead of Result
Inconsistent with all other fallible methods.
FIX: Return Result<(), CapabilityDeriveError> where a NotFound or InvalidId
variant signals that the ID was not valid.

HIGH | capabilities.rs | CapRights is an enum, should be a bitflag set
Using &[CapRights] for combining rights is awkward. A newtype over u8 with
associated constants enables direct bit operations.
FIX: Replace enum CapRights with:
  pub struct CapRights(u8);
  impl CapRights {
    pub const READ: Self = Self(1 << 0);
    pub const WRITE: Self = Self(1 << 1);
    // etc.
    pub const fn union(self, other: Self) -> Self { Self(self.0 | other.0) }
    pub const fn is_subset_of(self, other: Self) -> bool { (self.0 & !other.0) == 0 }
  }
The escalation check in mint_derived becomes:
  if !derived.rights.is_subset_of(parent.rights) { return Err(...); }

HIGH | capabilities.rs | CapObject monolithic enum couples all object types
Every new kernel object type requires editing CapObject. Consider an opaque
{ kind: u8, handle: u64 } pair instead, with semantics interpreted by the
subsystem owning that object type.

HIGH | capabilities.rs | generation skip-zero logic needs comment
  slot.generation = slot.generation.wrapping_add(1).max(1);
The .max(1) is doing load-bearing work silently.
FIX: Replace with explicit branch:
  let next = slot.generation.wrapping_add(1);
  slot.generation = if next == 0 { 1 } else { next };

MEDIUM | capabilities.rs | while loops instead of for ranges (recurring)
FIX: Same as ipc.rs — replace all while-counter loops with for range loops.

MEDIUM | capabilities.rs | Missing Display for CapabilityDeriveError
FIX: Implement core::fmt::Display.

MEDIUM | capabilities.rs | CapabilitySpace ~11KB — reducible
Removing name field saves ~2KB. Switching to opaque CapObject saves ~2KB more.

MODULE: scheduler.rs
================================================================================

CRITICAL | scheduler.rs | encode_task/decode_task off-by-one corrupts tid=u64::MAX
The +1/-1 sentinel trick encodes tid+1 to avoid NO_TASK=0. When tid=u64::MAX,
(u64::MAX + 1) wraps to 0 which decodes as None — the task silently disappears.
FIX: Use an explicit OCCUPIED_BIT instead of the +1 offset:
  const OCCUPIED_BIT: u64 = 1 << 63;
  const PRIORITY_SHIFT: u64 = 61;
  const TID_MASK: u64 = (1u64 << PRIORITY_SHIFT) - 1;
  fn encode_task(task: ScheduledTask) -> u64 {
    OCCUPIED_BIT | ((task.priority as u64) << PRIORITY_SHIFT) | task.tid
  }
  fn decode_task(encoded: u64) -> Option<ScheduledTask> {
    if encoded & OCCUPIED_BIT == 0 { return None; }
    // extract priority and tid directly, no +1/-1
  }
This also makes TID 0 a valid task ID.

CRITICAL | scheduler.rs | dequeue_highest silent None on invariant violation
  let mut best = decode_task(self.run_queue[self.head])?;
If self.run_queue[self.head] is NO_TASK despite self.len > 0, this returns None
silently instead of panicking on the violated invariant.
FIX: Use unwrap_or_else(|| panic!("scheduler invariant violated: len={} but head slot empty", self.len))

CRITICAL | scheduler.rs | on_preempt suppresses higher-priority tasks on queue-full
When enqueue fails during preemption, the current task is re-scheduled even if
higher-priority tasks are waiting in the queue.
FIX: If re-enqueue fails and the queue is non-empty, this is a kernel panic
condition (scheduler inconsistency), not a silent re-run.

CRITICAL | scheduler.rs | SmpScheduler owns current_cpu — wrong for SMP
current_cpu as a field of shared SmpScheduler means CPU 0 can corrupt CPU 1's
view of its own identity. current_cpu must be a CPU-local variable.
FIX: Remove current_cpu from SmpScheduler entirely. Pass CpuId explicitly to
all methods. Remove the convenience forwarders (dispatch_next, enqueue, etc.)
that use self.current_cpu. Each CPU calls the _on(cpu) variants with its own
locally-known CPU ID.

CRITICAL | scheduler.rs | dequeue_highest is O(N) with array compaction in hot path
The array compaction loop shifts all elements on every dispatch. This is O(N)
per preemption.
FIX: Replace the single run_queue ring with three per-priority rings:
  queues: [RingQueue; 3],  // index by TaskPriority as u8
enqueue is O(1), dequeue_highest scans 3 queues for first non-empty (O(1)),
array compaction disappears entirely.

HIGH | scheduler.rs | MAX_RUN_QUEUE power-of-two not asserted
FIX: Add: const _: () = assert!(MAX_RUN_QUEUE.is_power_of_two());

HIGH | scheduler.rs | duplicate enqueue silently drops priority update
If a task is enqueued with Normal priority then re-enqueued with High, the
second call returns Ok(()) but priority is not updated.
FIX: Either return Err(SchedulerError::AlreadyQueued) for duplicates, or
update the priority of the existing entry in-place.

HIGH | scheduler.rs | SchedulerError defined but never returned
All public methods return Result<_, u64> or Result<_, ()> instead of
Result<_, SchedulerError>.
FIX: Use SchedulerError in all return types. AlreadyQueued, QueueFull, InvalidCpu,
CpuOffline are all useful and already defined.

HIGH | scheduler.rs | Raw u64 for tid instead of ThreadId
FIX: Replace all tid: u64 parameters and Option<u64> returns with ThreadId.

HIGH | scheduler.rs | least_loaded_online_cpu biased toward CPU 0 on tie
FIX: Track a starting offset that rotates on each call to implement round-robin
tie-breaking instead of always preferring the lowest CPU ID.

HIGH | scheduler.rs | set_present_cpu_bitmap silently resets online CPU state
FIX: Document that this resets all CPU online state, or change it to only modify
the present bitmap without touching online status.

MEDIUM | scheduler.rs | while loops instead of for ranges (recurring)
MEDIUM | scheduler.rs | is_none_or MSRV concern — stable since Rust 1.82
FIX: Use best.map_or(true, |...| ...) if targeting older toolchains.
MEDIUM | scheduler.rs | TaskPriority ordering is counterintuitive (High=0)
FIX: Add a doc comment: "Lower numeric value = higher scheduling priority."
MEDIUM | scheduler.rs | contains_tid O(N) on every enqueue
FIX: Enforce uniqueness at call sites instead, removing the check from enqueue.

MODULE: smp.rs
================================================================================

CRITICAL | smp.rs | SpinLock without IRQ save causes deadlock from interrupt context
CrossCpuWorkQueue uses SpinLock which does not disable interrupts. If submit()
is called from an interrupt handler on a CPU that is already inside take(), the
CPU deadlocks.
FIX: Create a SpinLockIrq<T> wrapper that saves/restores interrupt flags on
lock/unlock. Use SpinLockIrq for CrossCpuWorkQueue.

CRITICAL | smp.rs | MAX_CROSS_CPU_WORK power-of-two not asserted
FIX: Add: const _: () = assert!(MAX_CROSS_CPU_WORK.is_power_of_two());

CRITICAL | smp.rs | target_cpu not validated at submit()
WorkItem carries target_cpu but submit() accepts any WorkItem without checking
whether target_cpu is online or valid.
FIX: Validate target_cpu against a topology reference at submit() time, or make
WorkItem constructors require a validated CpuId.

CRITICAL | smp.rs | Single shared queue is wrong routing model for SMP
A single CrossCpuWorkQueue means any CPU can drain work meant for another CPU.
FIX: Use per-CPU inboxes:
  pub struct SmpMailbox {
    inboxes: [CrossCpuWorkQueue; MAX_CPUS],
  }
  impl SmpMailbox {
    pub fn send_to(&self, cpu: CpuId, item: WorkItem) -> Result<(), WorkItem>
    pub fn drain_current(&self, cpu: CpuId) -> impl Iterator<Item=WorkItem>
  }
With per-CPU inboxes, target_cpu inside WorkItem becomes redundant — remove it.

HIGH | smp.rs | WorkItem conflates routing and payload
target_cpu inside each variant is routing metadata that belongs at the queue
level, not in the payload.
FIX: After per-CPU inbox refactor, simplify to:
  pub enum WorkItem { Reschedule, TlbShootdown { asid: Asid }, WakeTask { tid: ThreadId } }

HIGH | smp.rs | TlbShootdown too coarse — no VA range
FIX: Add optional VA range:
  TlbShootdown { asid: Asid, va_range: Option<(VirtAddr, VirtAddr)> }
This allows targeted sfence.vma instead of full ASID flush.

HIGH | smp.rs | pending() TOCTOU hazard
pending() acquires the lock then releases it. Any check-then-act on the result
is inherently racy. This must not be used in production logic paths.
FIX: Mark as #[cfg(debug_assertions)] or restrict to diagnostic/telemetry use only.
Alternatively, expose len as an AtomicUsize for lock-free diagnostic reads.

HIGH | smp.rs | Option<WorkItem> wastes space vs MaybeUninit
FIX: Replace ring: [Option<WorkItem>; N] with ring: [MaybeUninit<WorkItem>; N].
Use MaybeUninit::write on push, MaybeUninit::assume_init_read on pop.
Safety invariant: slots in [head, head+len) mod capacity are initialized.

MEDIUM | smp.rs | WorkQueue::len() shadows field name self.len
FIX: Rename the field to count or the method to item_count.
MEDIUM | smp.rs | Missing target_cpu() accessor on WorkItem
FIX: Add fn target_cpu(&self) -> CpuId { match self { ... } }

MODULE: lock.rs
================================================================================

CRITICAL | lock.rs | lock() missing #[must_use]
  pub fn lock(&self) -> SpinLockGuard<'_, T>
A caller writing self.lock.lock(); immediately releases the lock and enters the
protected region with no exclusion. The compiler gives no warning.
FIX: Add #[must_use = "if unused, the lock is immediately released when the guard is dropped"]
to lock().

CRITICAL | lock.rs | Outer CAS retry path missing spin_loop() hint
When compare_exchange_weak fails spuriously, the outer loop retries the CAS
without calling spin_loop(). This burns pipeline resources without a hint.
FIX: Add spin_loop() after the inner spin exits:
  while self.held.compare_exchange_weak(...).is_err() {
    while self.held.load(Relaxed) { spin_loop(); }
    spin_loop();  // hint on CAS retry
  }

CRITICAL | lock.rs | No IRQ-save variant — deadlock risk from interrupt context
The doc comment acknowledges this but provides no enforcement. All subsystems
using SpinLock for interrupt-accessible resources are at risk.
FIX: Add SpinLockIrq<T> that saves/restores RFLAGS (x86) or sstatus.SIE (RISC-V)
on lock/unlock. The guard's Drop must restore IRQ state after releasing the lock.
Note: restore IRQs AFTER releasing the spinlock, not before.

HIGH | lock.rs | No cache-line padding between held and value
held: AtomicBool and value: UnsafeCell<T> share a cache line. When the protected
value is written, the cache line is invalidated on all waiting CPUs, forcing
them to re-fetch just to spin on held.
FIX: Pad held to its own cache line:
  pub struct SpinLock<T> {
    held: AtomicBool,
    _pad: [u8; 63],
    value: UnsafeCell<T>,
  }
Or use #[repr(align(64))] on a wrapper struct around held.

HIGH | lock.rs | No debug-mode recursive lock detection
A thread acquiring the same lock twice deadlocks silently in release builds.
FIX: In debug builds, add owner_cpu: AtomicU8 field. On lock(), assert that
owner_cpu != current_cpu_id(). Reset to sentinel on drop.

MEDIUM | lock.rs | Relaxed failure ordering on try_lock CAS needs comment
FIX: Add comment: "Failure ordering is Relaxed: a failed CAS acquires nothing,
so no memory ordering guarantee is needed on the failure path."

MEDIUM | lock.rs | PhantomData choice needs comment
FIX: Add comment on _not_send field:
"!Send: a guard must be released on the CPU/thread that acquired the lock.
*const UnsafeCell<()> is !Send + !Sync without affecting other bounds."

MEDIUM | lock.rs | No exponential backoff under high contention
FIX: In the inner spin loop, double the spin_loop() count on each retry up to
a MAX_BACKOFF constant. Reset on successful acquisition.

LOW | lock.rs | Debug impl for SpinLockGuard should explain why T is omitted
FIX: Add comment: "T is intentionally not printed: its Debug impl may acquire locks."

MODULE: syscall.rs
================================================================================

CRITICAL | syscall.rs | Unvalidated user pointer/len encoded into shared-mem region
In handle_ipc_send, user_ptr_or_offset and len come directly from frame.args
(raw userspace register values) and are encoded into a SharedMemoryRegion
without any bounds checking. A malicious sender can encode arbitrary kernel
addresses.
FIX: Add before encoding:
  fn validate_user_region(offset: u64, len: u64) -> Result<(), SyscallError> {
    let end = offset.checked_add(len).ok_or(SyscallError::InvalidArgs)?;
    if end > USER_ADDR_MAX { return Err(SyscallError::InvalidArgs); }
    Ok(())
  }
Call this before constructing SharedMemoryRegion.

CRITICAL | syscall.rs | Shared-mem detection via payload-length heuristic
In handle_ipc_recv:
  if msg.transferred_cap().is_some() && msg.as_slice().len() == SharedMemoryRegion::ENCODED_LEN
A malicious sender can construct any 16-byte message with a cap attached and
cause the receiver to misinterpret it as a SharedMemoryRegion.
FIX: Use the Message::opcode field as a discriminant. Define:
  pub const OPCODE_INLINE:     u16 = 0;
  pub const OPCODE_SHARED_MEM: u16 = 1;
The kernel sets opcode based on which send path was taken. The recv path
dispatches on opcode, not payload length.

CRITICAL | syscall.rs | No capability rights check before IPC send/recv
handle_ipc_send calls kernel.ipc_send(cap, msg) without checking CapRights::Send
on the capability. handle_ipc_recv does not check CapRights::Receive. Any thread
that can name an endpoint's CapId can send to or receive from it.
FIX: In handle_ipc_send, before calling ipc_send:
  let endpoint_cap = kernel.cspace.get(cap).ok_or(SyscallError::InvalidCapability)?;
  if !endpoint_cap.has_right(CapRights::Send) { return Err(SyscallError::MissingRight); }
Similarly for handle_ipc_recv with CapRights::Receive.

CRITICAL | syscall.rs | transfer_cap_arg treats raw == 0 as no-cap sentinel
CapId(0) can be a valid capability (slot 0 with non-zero generation). Using 0
as a second sentinel alongside u64::MAX is fragile.
FIX: Use only SYSCALL_NO_TRANSFER_CAP (u64::MAX) as the sentinel. Document
why 0 is safe to exclude if that remains the case, or remove the || raw == 0 check.

CRITICAL | syscall.rs | Syscall layer directly matches CapObject variants
The dispatch code pattern-matches on CapObject::MemoryObject and CapObject::DmaRegion
directly. This couples the ABI to internal object representation.
FIX: Move object-type checks into methods on KernelState:
  kernel.validate_memory_grant_cap(cap)?
  kernel.validate_ipc_send_cap(cap)?
The syscall layer calls these methods; the object type logic lives in KernelState.

HIGH | syscall.rs | sender_tid_to_ret fails on 32-bit for high TIDs
usize::try_from(tid: u64) fails if tid > u32::MAX on 32-bit targets, causing
Internal error after successfully dequeuing the message (which is then lost).
FIX: Either restrict TIDs to usize range, or split sender TID across two return
registers for 32-bit targets.

HIGH | syscall.rs | Tautological compile-time asserts catch nothing
  const _: [(); SYSCALL_COUNT] = [(); 3];  // SYSCALL_COUNT IS 3
FIX: Replace with:
  const _: () = assert!(SYSCALL_COUNT == Syscall::VARIANT_COUNT);
  const _: () = assert!(SYSCALL_ARG_TRANSFER_CAP < syscall_abi::TRAPFRAME_ARG_REGS);

HIGH | syscall.rs | SYSCALL_NO_TRANSFER_CAP and ipc.rs NO_TRANSFER_CAP are duplicated
FIX: Define once in a shared ABI module. In syscall.rs:
  pub const SYSCALL_NO_TRANSFER_CAP: u64 = crate::kernel::ipc::NO_TRANSFER_CAP;

HIGH | syscall.rs | handle_ipc_recv writes back to frame.args (input registers)
frame.args[SYSCALL_ARG_INLINE_PAYLOAD0] is overwritten as a return value. If
the ABI doesn't define these as input/output registers, this silently corrupts
caller-saved state.
FIX: Add TrapFrame methods:
  fn set_ret_payload(&mut self, words: [usize; 2])
  fn ret_payload(&self) -> [usize; 2]
Make the ABI contract explicit in TrapFrame's API.

HIGH | syscall.rs | opcode always set to 0 — field is dead or misused
FIX: Use opcode to distinguish message types (see shared-mem heuristic fix above).
Define OPCODE_INLINE=0, OPCODE_SHARED_MEM=1, OPCODE_TASK_EXIT=0xEE, etc.

MEDIUM | syscall.rs | handle_ipc_send too long with duplicated flag logic
The flag/cap computation is duplicated across the large-payload and small-payload branches.
FIX: Extract:
  fn build_message(sender_tid: u64, transfer_cap: Option<CapId>, payload: &[u8])
    -> Result<Message, SyscallError>

MEDIUM | syscall.rs | Raw frame.args[N] indexing throughout
FIX: Add TrapFrame accessor methods:
  fn syscall_cap_arg(&self) -> u64
  fn syscall_ptr_arg(&self) -> usize
  fn syscall_len_arg(&self) -> usize

MODULE: task.rs
================================================================================

CRITICAL | task.rs | ThreadControlBlock derives Copy
Copy on a TCB means any function taking it by value silently makes a bitwise
duplicate. The TCB IS the thread's identity — there must be exactly one
authoritative copy. With Copy, mutations to a local copy are silently discarded.
FIX: Remove Copy from ThreadControlBlock. Keep Clone only if needed for
explicit snapshot operations (e.g., debug dumps). Review all fields that derive
Copy solely for TCB embeddability.

CRITICAL | task.rs | TaskStatus::Running not enforced by scheduler
Anyone with &mut ThreadControlBlock can write status = TaskStatus::Running
without going through the scheduler, creating state divergence.
FIX: Either:
  Option A: Make status private, add transition methods that validate edges.
  Option B: Remove Running from TaskStatus entirely. Derive "running" from
    scheduler state: a task is running iff it is current on some CPU.
Option B is architecturally correct for a microkernel.

CRITICAL | task.rs | POSIX-layer fields in kernel TCB
tls_restore_pending (TLS management), robust_futex (Linux robust futex lists),
brk_base/brk_end (heap pointer for sbrk) are POSIX/libc concerns.
FIX: Remove robust_futex, brk_base, brk_end from ThreadControlBlock.
Reduce TLS to a single tls_ptr: Option<VirtAddr> for the context-switch register.
Robust futex cleanup on task death should be a message to the POSIX server.
brk management belongs in the allocator/POSIX server.

CRITICAL | task.rs | WaitReason::Futex(usize) carries raw kernel virtual address
FIX: Change to WaitReason::Futex(VirtAddr). Validate before use.

HIGH | task.rs | brk_base and brk_end as separate Option<VirtAddr> fields
Nothing prevents brk_base=None with brk_end=Some(x), or brk_end < brk_base.
FIX: Combine into a single:
  pub struct BrkRegion { pub base: VirtAddr, pub end: VirtAddr }
  impl BrkRegion {
    pub fn new(base: VirtAddr, end: VirtAddr) -> Option<Self> {
      if end.0 >= base.0 { Some(Self { base, end }) } else { None }
    }
  }
  pub brk: Option<BrkRegion>,

HIGH | task.rs | RestartState::budget: u8 has no safe accessor
FIX: Add:
  impl RestartState {
    pub fn can_restart(&self) -> bool { self.budget > 0 }
    pub fn consume_restart(&mut self) -> bool {
      if self.budget == 0 { return false; }
      self.budget -= 1;
      true
    }
  }

HIGH | task.rs | WaitReason::Join(u64) should use ThreadId
FIX: Change to WaitReason::Join(ThreadId).

HIGH | task.rs | thread_group_id: u64 should be ThreadGroupId newtype
FIX: Add pub struct ThreadGroupId(pub u64); and use it.

HIGH | task.rs | TaskClass not tied to capability system
FIX: Document explicitly which capabilities are granted to each TaskClass at
creation time, or remove TaskClass and rely purely on held capabilities.

HIGH | task.rs | FaultPolicy imported from bootstrap module
Runtime task state depending on bootstrap types creates coupling.
FIX: Move FaultPolicy definition to task.rs or a shared policy.rs module.

MEDIUM | task.rs | tls_base, user_entry, user_stack_top typed as usize not VirtAddr
FIX: Change all address-valued usize fields to VirtAddr.

MEDIUM | task.rs | last_exit_code: Option<u64> ambiguous on None
FIX: Encode exit code in status: TaskStatus::Exited(u64) so it is impossible
to have Exited status without an exit code.

MEDIUM | task.rs | No constructor for ThreadControlBlock
FIX: Add ThreadControlBlock::new(tid: ThreadId, class: TaskClass, asid: Option<Asid>)
with documented defaults for all other fields.

MODULE: timer.rs
================================================================================

CRITICAL | timer.rs | tick() and should_preempt() are order-dependent
should_preempt() has hidden precondition: must be called after tick(). The
deduplication via last_preempt_tick exists only to paper over independent calls.
FIX: Make tick() and should_preempt() private or remove them. Make
tick_and_check() the only public interface.

CRITICAL | timer.rs | Tick(0) silently skipped as boundary after wrapping
  let at_boundary = self.current.0 != 0 && ...
After u64 wraps to 0, that boundary tick is skipped.
FIX: Replace the modulo approach with a countdown counter (see performance fix
below), which naturally handles wrap-around correctly.

HIGH | timer.rs | Modulo division on every tick — use countdown counter
  self.current.0 % self.quantum_ticks
Division is expensive on embedded targets without hardware dividers.
FIX: Replace Timer with a countdown design:
  struct Timer {
    current: Tick,
    quantum_ticks: u64,
    ticks_remaining: u64,
  }
  fn tick_and_check(&mut self) -> (Tick, bool) {
    self.current.0 = self.current.0.wrapping_add(1);
    self.ticks_remaining -= 1;
    let preempt = self.ticks_remaining == 0;
    if preempt { self.ticks_remaining = self.quantum_ticks; }
    (self.current, preempt)
  }
This is O(1), has no division, fixes the wrap-around skip, and eliminates
last_preempt_tick stale state entirely.

HIGH | timer.rs | Tick and TickInstant from task.rs are the same type defined twice
FIX: Define once in a shared time.rs module:
  pub struct Tick(pub u64);       // absolute instant

  fn drain_for_console<F>(&self, from_seq: u64, sink: F) -> (usize, u64)
Each console driver maintains its own cursor independently.

HIGH | printk.rs | StackBuf::write_str silently truncates and returns Ok(())
When the buffer is full, write_str copies nothing and returns Ok(()). The
fmt::Write machinery thinks the write succeeded, so format_args may produce
a silently truncated message with no indication of overflow.
FIX: Track truncation:
  fn write_str(&mut self, s: &str) -> fmt::Result {
    let bytes = s.as_bytes();
    let avail = PRB_MSG_MAX.saturating_sub(self.len);
    let copy = core::cmp::min(avail, bytes.len());
    self.buf[self.len..self.len + copy].copy_from_slice(&bytes[..copy]);
    self.len += copy;
    if bytes.len() > avail {
      self.truncated = true;  // add this field
    }
    Ok(())
  }
If truncated, append "[truncated]" marker in printk_args before pushing to ring.

HIGH | printk.rs | drain_for_console does not update reader_seq atomically on panic
If sink() panics, reader_seq is not updated and the same messages are re-delivered
on the next drain call. In a kernel, panicking in a log drain is a real scenario
(e.g., console write_line encounters a hardware fault).
FIX: Update reader_seq after each individual message rather than at the end:
  self.reader_seq.store(seq + 1, Ordering::Release);  // inside the loop
This makes progress unconditionally, even if the loop is interrupted.

MEDIUM | printk.rs | read_if_seq memory ordering may allow torn reads
The seqlock pattern in read_if_seq checks committed and seq before and after
reading bytes. But if a writer begins after the first committed check and
the slot is being written with the same seq (impossible with monotonic seqs
but worth hardening), the reader could see partially written data.
FIX: Add a compiler fence between the initial committed/seq check and the
byte reads to prevent the compiler from reordering:
  core::sync::atomic::fence(Ordering::Acquire);
  // read bytes here
  core::sync::atomic::fence(Ordering::Acquire);
  // check seq1 and committed again

MEDIUM | printk.rs | while loops in write() and read_if_seq() should be for ranges
  while i < len { self.bytes[i].store(...); i += 1; }
FIX: Replace with for i in 0..len throughout.

MEDIUM | printk.rs | Level/context u8 → enum conversion uses manual match in read_if_seq
FIX: Implement TryFrom<u8> for LogLevel and PrintkContext:
  impl TryFrom<u8> for LogLevel {
    type Error = ();
    fn try_from(v: u8) -> Result<Self, ()> { match v { 0 => Ok(Self::Emerg), ... } }
  }
Use level.try_from(...).unwrap_or(LogLevel::Debug) in read_if_seq.

MEDIUM | printk.rs | PrintkRecord is Copy and ~208 bytes — expensive copies
Each snapshot_latest call copies records of ~208 bytes by value into the output
slice. In a hot diagnostic path this adds up.
FIX: Document the size in a comment. If records are frequently copied, consider
a read-into-borrowed-slice API instead of returning by value.

MEDIUM | printk.rs | threaded_drain_applies_console_loglevel test is stateful
The test calls set_console_loglevel(LogLevel::Warn) on the global static PRINTK.
Concurrent tests will see a modified log level.
FIX: Tests that modify global state must either run sequentially (#[serial]) or
use a local PrintkRing instance rather than the global static.

LOW | printk.rs | printk_args is not marked inline
On hot paths (frequent logging), the format_args overhead is fixed but the
function call overhead adds up. Consider #[inline] on printk_args.

LOW | printk.rs | PRINTK static ring ~26KB in BSS — should be documented
128 slots × (8+2+1+1+1 + 192) bytes = ~26KB. Fine for a kernel, but document it.

CROSS-1 | RECURRING: while loops instead of for ranges
Every module uses while i < N { ...; i += 1 }

CROSS-2 | RECURRING: Missing Display impls for error types
IpcError, CapabilityDeriveError, SchedulerError, VmError, KernelError, TopologyError
all lack core::fmt::Display.
FIX: Implement Display for all error types with human-readable messages.

CROSS-3 | RECURRING: Power-of-two ring buffer sizes unasserted
ipc.rs, scheduler.rs, smp.rs, printk.rs all use bitmask/modulo without asserting
that capacity is a power of two.
FIX: Add to each module:
  const _: () = assert!(CAPACITY.is_power_of_two(), "must be power of two");

CROSS-4 | RECURRING: Raw u64/usize for typed identifiers
tid: u64, thread_group_id: u64, irq: u16 instead of ThreadId, ThreadGroupId, IrqNumber.
FIX: Define newtypes for every domain identifier and use them consistently.

CROSS-5 | RECURRING: Capability name field
name: &'static str in Capability wastes 2KB across 128 slots.
FIX: Remove name from Capability struct and all Capability::new() call sites.

CROSS-6 | ARCHITECTURE: Tick/TickInstant/TickDuration defined in multiple places
timer.rs defines Tick, task.rs defines TickInstant and TickDuration independently.
FIX: Create time.rs with pub struct Tick(pub u64) and pub struct TickDuration(pub u64).
Import everywhere. Remove duplicates.

CROSS-7 | ARCHITECTURE: Copy on kernel identity types
ThreadControlBlock and TrapFrame derive Copy — these represent unique kernel
objects and must not be silently duplicated.
FIX: Remove Copy from both. Audit all call sites.

CROSS-8 | ARCHITECTURE: Bring-up protocol requires topology and scheduler coordination
The SMP bring-up sequence spans CpuTopology, SmpScheduler, and KernelState.
There is no single authoritative sequence document.
FIX: Add a module-level doc comment in bootstrap.rs or a separate smp_bringup.rs
documenting the exact sequence:
  1. Bootstrap calls topology.start_secondary_cpu(cpu)
  2. Bootstrap sends SIPI/platform-specific wake signal to secondary
  3. Secondary entry runs, calls topology.acknowledge_secondary_cpu(self_id)
  4. Bootstrap spins on topology.check_secondary_ack(cpu) until Ok
  5. Bootstrap calls topology.mark_cpu_online(cpu)
  6. Bootstrap enqueues initial task on secondary via scheduler.enqueue_on(cpu, tid)
